## Specific scrutiny (per the R2 prompt)

- **Chain-replay-on-read in `from_dict`** — solid. Constructing the `Log` empty and replaying every entry through `append` means corrupt parent_id, iteration-skip, duplicate-id, and append-after-resolved all surface on read. The `data.get("entries") or []` defaults a missing/null entries list to `[]` cleanly. One residual edge: if the YAML's `entries` field is, say, an int, `for entry_data in <int>` raises `TypeError`, not `LogSchemaError` — minor, not worth blocking.
- **Path-traversal defense in `TextFileAdapter`** — the `relative_to` check is the right primitive in 3.11+. I verified:
  - `"../sibling"`, `"nested/sub"`, `"back\\slash"` namespaces rejected (separator check).
  - `".hidden"` namespace rejected (dot-prefix check).
  - `""` namespace rejected.
  - `load("../sibling.txt")` rejected via the `relative_to(base_resolved)` check.
  - **Absolute `output_ref="/etc/passwd"` rejected**: `(self._base / "/etc/passwd").resolve()` collapses to `/etc/passwd` (Path's documented behavior — joining an absolute path discards the left side), which then fails `relative_to(base_resolved)`. Confirmed empirically. Good.
  - Symlink behavior: `Path.resolve()` follows symlinks, so a symlink *inside* base_dir pointing outside would be detected (resolved path falls outside base). A symlink whose target ends back inside base_dir is allowed, which is the right call for normal use.
- **SemVer version handling** — handles the documented happy path correctly. Edge cases worth flagging as a minor (see N1 below): negative integers, leading zeros, and whitespace-padding are accepted by `int()`. Not exploitable (negative major fails the major-equality check downstream), but loose for a value baked into the public file format.
- **`parent_session_id` round-trip** — `to_dict` omits the key when None (verified by `test_parent_session_id_omitted_when_none`); `from_dict` uses `data.get("parent_session_id")` which handles both omitted and explicit-null cases (verified by direct probe of YAML containing `parent_session_id: null`). Clean.
- **`Loop` session_id+log conflict check** — the guard is `if session_id is not None and session_id != log.session_id`, which correctly:
  - Permits `session_id=None` + `log` (auto-uses `log.session_id`).
  - Permits `session_id="X"` + `log.session_id="X"` (redundant but allowed; `test_resume_with_matching_session_id_is_allowed` confirms).
  - Raises on `session_id="X"` + `log.session_id="Y"`.
  - Crucially: when `log` is None, the auto-UUID4 fallback (`session_id or str(uuid4())`) is unaffected.
- **Duplicate-id check ordering** — duplicate-id check runs *after* parent_id mismatch and iteration mismatch. Order is fine: the test constructs a non-pathological case (entry at iteration 4 with `parent_id="c"` and `id="a"`, where "a" already exists at iteration 1) precisely so the prior checks pass and the duplicate-id check is what fires. For pathological inputs (e.g., entry where id == prev.id and parent_id == prev.id), the parent_id check fires first because it's the same value → matches → passes → then iteration → then duplicate. Error message on a "duplicate that's also a self-parent" reads as `parent_id mismatch` rather than `duplicate`, but normal flow can't construct one (your own id is unknown to your own parent_id). Not worth changing.

## New findings

### N1 — SemVer parser permissive on negatives, leading zeros, whitespace

**Severity:** minor
**File:** `src/cdd/log.py:170-190` (`_parse_schema_version`)
**Issue:** `_parse_schema_version` uses `int(parts[0])` / `int(parts[1])`, which accepts:
- Negative integers: `"-1.0"` → `(-1, 0)`, `"1.-1"` → `(1, -1)`
- Leading zeros: `"01.0"` → `(1, 0)`
- Whitespace padding: `" 1.0"` and `"1.0 "` → `(1, 0)`

None of these are security or correctness bugs — a negative-major value just fails the downstream `file_major != expected_major` check with a "unsupported version" error, and leading-zero / whitespace inputs round-trip to a numerically-equivalent version. But for a value persisted into the public file format, the parser is looser than the docstring implies (`MAJOR.MINOR` strongly suggests "non-negative integers, no leading zeros, no whitespace").

**Recommended fix:** Use a regex like `^(0|[1-9]\d*)\.(0|[1-9]\d*)$` (the standard SemVer `numeric identifier` rule) and document the strict shape. Doesn't need to ship in v0.1.0 — fold into v0.1.5 or v0.2.0 alongside the broader schema-versioning work.

## Strengths in the fix-fold

- **Test design is precise.** The duplicate-id test deliberately constructs the duplicate at iteration 4 with a *valid* parent_id pointer to "c" — so the parent_id check passes, the iteration check passes, and only the new duplicate-id check can be what fires. This is the right way to test a guard in isolation, not a happy-path negation.
- **`test_from_dict_replays_chain_validation_*` mutates a *valid* `to_dict` output post-serialization** — ensures the bug being caught is exactly the YAML-hand-edit corruption case, not a synthetic fixture that wouldn't survive serialization.
- **`_LOG_SCHEMA_VERSION` going underscore-private** is the right call. Keeping it symmetric with private `to_dict`/`from_dict` (F11 deferral) gives a coherent "consumers use `read_log`/`write_log`; tooling reaches in" contract.
- **`parent_session_id` omit-when-None** is the correct YAML aesthetic — the common case stays clean. The lineage-vs-chain distinction (cross-session lineage vs in-session chain) is documented in the field's docstring.
- **Forward-compat warning on higher MINOR** uses `warnings.warn(..., stacklevel=3)` and is verified with `pytest.warns(UserWarning, match="silently dropped")`. The stacklevel choice (3) correctly attributes the warning to the caller of `from_dict`, not to internal frames.
- **Path-traversal defense uses `Path.resolve()` + `relative_to()`** — the canonical pattern. Catches both `..` traversal and absolute-path injection in one check. Empirical verification of the absolute-path case is non-obvious (Python's path-join semantics) and the implementation gets it right.

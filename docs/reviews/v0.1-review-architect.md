# CDD v0.1.0 — Round 1 Architect Review

## Verdict: APPROVE-WITH-CHANGES

The architecture is fundamentally sound. The Adapter/Generator split is the right cleavage, the Loop's state machine encodes the discipline at the type level, and the Log's append-only chain is exactly what the methodology promises. The package is honest about being pre-alpha and the README/types match.

That said, four extensibility cliffs are visible from where v0.1.0 sits — they are cheap to address now while the surface has zero external consumers, and expensive to address once Anthropic generators (v0.2.0), the CLI (v0.3.0), and the FATHOMS vegetation adapter (v0.5.0) are pulling on the protocol. The most important is the `generate(prompt) -> Any` signature, which forces multi-modal and async/streaming generators into v2.0 territory unless caught in the next sprint or two.

None of the findings rise to "redesign required." All can be folded into the v0.2.0 sprint window.

## Detailed findings

### F1 — Generator signature is text-shaped

**Severity:** major
**Area:** protocol-design

**Issue:** `Generator.generate(prompt: str, *, seed, extra_params) -> Any` is a clean shape for a v0.1.0 text generator. By v0.2.0 it does not survive contact with reality:

- **Multi-modal.** Anthropic Claude with image-in / text-out: the `prompt` is no longer a string. `extra_params` is the wrong place to smuggle PIL Images — that path has no type discipline and the YAML log loses the input.
- **Structured prompts.** Claude has system/user/assistant/tool-use roles. Putting them all into one string loses provenance.
- **Streaming.** `-> Any` admits a generator-of-tokens, but the Loop's `propose()` calls `adapter.persist(raw_output)` synchronously — a token stream cannot persist incrementally without a different shape.
- **Async.** Once one provider goes async, the Loop's sync `propose()` becomes a fork point. Better to pick a stance now.

**Why it matters:** The fix is cheap with zero external consumers; expensive once Anthropic + OpenAI + fal.ai generators ship. The roadmap's S3 (v0.2.0) is "Anthropic generator implementation" — that sprint will discover this finding the hard way.

**Recommended fix:**

```python
@dataclass(frozen=True)
class GenerationRequest:
    prompt: str | list[Message]      # union admits multi-modal later
    seed: int | None = None
    extra_params: Mapping[str, Any] = ...

@dataclass(frozen=True)
class GenerationResult[T]:
    output: T
    model_used: ModelIdentity        # may differ from generator's claimed identity (fallback)
    raw_metadata: dict[str, Any]     # provider's response metadata: token counts, finish_reason, etc.

class Generator(Protocol[T]):
    def generate(self, request: GenerationRequest) -> GenerationResult[T]: ...
```

Adapter becomes `Adapter[T]` correspondingly. `EchoGenerator` becomes `Generator[str]`; a hypothetical `FluxGenerator` becomes `Generator[bytes]`. Async/streaming variants get sibling protocols (`AsyncGenerator`, `StreamingGenerator`) that compose cleanly.

If this is too much for v0.2.0, *at minimum* introduce `GenerationResult` so Anthropic's token counts and finish_reason aren't lost on the floor.

### F2 — Schema version as integer with strict equality

**Severity:** major
**Area:** log-schema

**Issue:** `LOG_SCHEMA_VERSION = 1` plus the `if version != LOG_SCHEMA_VERSION: raise` check means **any** schema change forces a major bump and cuts off backward read. Real schema evolution is mostly additive: adding `parent_session_id` (F12), adding `raw_metadata` (F1), adding `negotiated_determinism_tier` (F5).

**Why it matters:** Once external consumers pin a CDD version and accumulate logs, every schema bump means a migration tool. SemVer `MAJOR.MINOR` (where adds are MINOR-bumps and reads are forward-tolerant) is the standard answer; integer-only forecloses on it.

**Recommended fix:**

```python
LOG_SCHEMA_VERSION = "1.0"  # MAJOR.MINOR

def from_dict(data):
    version = str(data.get("cdd_log_version", ""))
    major, _, _ = version.partition(".")
    if major != LOG_SCHEMA_VERSION.split(".")[0]:
        raise LogSchemaError(...)
    if version != LOG_SCHEMA_VERSION:
        warnings.warn(f"log schema {version} differs in MINOR from {LOG_SCHEMA_VERSION}; reading forward-compatibly")
    # tolerant key reads...
```

Document: MAJOR bump = breaking; MINOR bump = additive; CDD reads same-MAJOR forward.

### F3 — output_ref as str doesn't fit large in-memory artifacts

**Severity:** major
**Area:** dogfood-readiness

**Issue:** The vegetation adapter (S6) outputs a 4096×4096 float32 mask = 64 MB per iteration. A 100-iteration session is 6.4 GB of disk if every `persist()` call writes immediately. Worse: the human reviews mask N before deciding whether to even look at mask N+1, and may want to compare mask N vs mask N-2 visually. Forcing every candidate to disk before review is wasteful.

The current shape gives the adapter no way to say "I'll keep this in memory until you ACCEPT, then write."

**Why it matters:** First real adopter (FATHOMS, S6/v0.5.0) hits this immediately. Workaround would be "write to /tmp, delete on REJECT" which leaks state if the process dies.

**Recommended fix:** Two options, in increasing scope:

1. **Cheap.** Document that `persist` may be a no-op for in-memory adapters and `output_ref` may be an opaque token (not a file path). Add `Adapter.materialize(output_ref) -> Any` (default = `load`) for replay tooling that needs the actual bytes back.

2. **Right.** Split persistence in two: `Adapter.stage(raw) -> output_ref` (in-memory or temp) and `Adapter.commit(output_ref) -> str` (called only by Loop on ACCEPT). Loop's `review(REJECT)` calls `Adapter.discard(output_ref)`. This generalizes to fal.ai: `stage = remember URL`, `commit = download bytes`.

Option 2 is the right shape for v0.5.0; option 1 unblocks dogfooding without committing.

### F4 — Loop forbids batch propose

**Severity:** major
**Area:** loop-state-machine

**Issue:** The Loop's docstring is direct: "A Loop holds at most one pending proposal at a time. You must review() a pending proposal before issuing the next propose()." That is exactly right for *one critique iteration at a time* — but the methodology's actual pitch ("AI generates at volume, human reviews each iteration") often manifests as:

- "Generate 4 vegetation masks at the same prompt+seed-spread, show me the grid, I pick one."
- "Generate 12 dialogue line variants, I rank them."
- "Generate 8 building variations across the kit, I assign each to a building slot."

The Loop offers no first-class shape for this. The workaround — `propose(); review(REJECT); propose(); review(REJECT); propose(); review(ACCEPT)` — drops the spatial relationship between candidates (replay just sees "rejected 3, accepted 1" with no signal that they were a *batch*).

**Why it matters:** Vegetation adapter (S6) is exactly this workflow. The CLI's `cdd review` interactive loop (S4) almost certainly wants a "show me 4, pick one" mode.

**Recommended fix:** Either:

1. **Lightweight.** Document the batch pattern as "N back-to-back REJECTs followed by an ACCEPT, all carrying the same prompt and a shared `batch_id` in `extra_params`." Cheap; keeps the protocol tight.

2. **First-class.**

```python
def propose_batch(self, prompt: str, n: int, *, seeds: list[int] | None) -> list[Candidate]: ...
def review_batch(self, accepted: Candidate, rejected: list[Candidate], *, notes: str = "") -> list[LogEntry]: ...
```

Each batch logs N entries (N-1 REJECTs + 1 ACCEPT) with a shared `batch_id`. Replay is unchanged; analysis tools can group by batch_id.

I'd ship option 1 as documentation in v0.1.0 (no code change) and treat option 2 as a v0.3.0 ticket once the CLI surfaces the pattern.

### F5 — Determinism tier on Adapter, but the Generator decides

**Severity:** major
**Area:** determinism

**Issue:** `TextFileAdapter.determinism_tier = SEMANTIC_B` is a coherent default for a content-addressed disk store. But the actual reproducibility of a session depends on the generator: pair this same adapter with `EchoGenerator` (deterministic) and you can honestly claim STRICT_A; pair it with a hypothetical `OpenAIGenerator` (which doesn't honor seeds) and SEMANTIC_B is the most you can promise. The adapter doesn't know which generator it's paired with — the Loop does.

The honest tier is `min(adapter_capability, generator_capability, user_request)`. The Log records the *negotiated* tier, not a unilateral declaration.

**Why it matters:** v0.6.0 (S7) is "Determinism enforcement" — replay tooling that takes a Log with `determinism_tier = STRICT_A` and refuses to replay if the recorded model is unavailable. If the recorded tier was a unilateral adapter claim that was never true (because the generator didn't support seeds), enforcement enforces a lie. This is the kind of design hole that compounds.

**Recommended fix:**

```python
class Adapter(Protocol):
    @property
    def max_tier(self) -> DeterminismTier: ...   # most strict tier this adapter CAN honor

class Generator(Protocol):
    @property
    def max_tier(self) -> DeterminismTier: ...   # most strict tier this generator CAN honor

# Loop:
def __init__(self, *, adapter, generator, requested_tier: DeterminismTier = DeterminismTier.SEMANTIC_B):
    self._tier = _negotiate(adapter.max_tier, generator.max_tier, requested_tier)
    # Log records self._tier
```

`_negotiate` picks the *strictest* tier all three permit (with STRICT_A < SEMANTIC_B < BEST_EFFORT_C as a chain — strictest is leftmost). If user requests STRICT_A but generator only supports SEMANTIC_B, raise; users opt down explicitly, never silently up.

### F6 — `LOG_SCHEMA_VERSION` shouldn't be public

**Severity:** moderate
**Area:** API-surface

**Issue:** Exporting the integer `LOG_SCHEMA_VERSION` as a top-level public symbol invites callers to `if cdd.LOG_SCHEMA_VERSION >= 2: ...`, which becomes the API contract you didn't mean to ship.

**Why it matters:** Per F2, the version itself should change shape (int → str). Exporting it makes that a breaking change to *every* downstream that reads it.

**Recommended fix:** Underscore-prefix internally; expose via `cdd.log.schema_version()` if needed, returning a structured `(major, minor)` tuple. Better: don't expose at all in v0.1.0; add the function in v0.6.0 when migration tooling needs it.

### F7 — No protocol-conformance test scaffolding

**Severity:** moderate
**Area:** API-surface / dogfood-readiness

**Issue:** Adapter and Generator are Protocols. The package ships no test helpers for "verify this object conforms." Anyone writing a `VegetationDensityAdapter` needs a test that catches "I forgot to return the tier as the right type." Today's only signal is `isinstance(obj, Adapter)` (true thanks to `@runtime_checkable`), which only checks attribute existence, not behavior.

**Why it matters:** The point of dogfooding is to catch protocol holes. Without conformance harnesses, the FATHOMS adapter will encode tribal knowledge that doesn't transfer to the next adopter.

**Recommended fix:** v0.2.0 ticket. Add `cdd.testing` with:

```python
def assert_adapter_conforms(adapter: Adapter, sample_input: Any) -> None: ...
def assert_generator_conforms(generator: Generator, sample_prompt: str) -> None: ...
class InMemoryAdapter:  # for downstream unit tests
    ...
class FixedGenerator:  # for downstream unit tests, takes a list of prompted -> output mappings
    ...
```

Document in the README that downstream adapters MUST pass `assert_adapter_conforms`.

### F8 — YAML scaling cliff

**Severity:** moderate
**Area:** log-schema

**Issue:** YAML for an N-entry log is `O(N)` parse time and you must parse it all to read the last entry. For text adapter logs (target use case in v0.1.0), N is small and prompts are short — fine. For vegetation logs at 100 iterations with multi-paragraph prompts and big extra_params, parsing dominates.

**Why it matters:** S6 dogfood will measure this. The methodology promise "log is human-readable and replayable" is preserved by YAML; the methodology promise "iterate fast" is not when the log itself is slow to load.

**Recommended fix:** No change v0.1.0 — flag in roadmap v0.4.0+ as "JSONL append mode." `read_log` auto-detects format via first line. `write_log` keeps current YAML behavior; an `append_log(path, entry)` helper writes JSONL incrementally without re-serializing prior entries.

### F9 — Synchronous, eager persistence

**Severity:** moderate
**Area:** dogfood-readiness

**Issue:** `Adapter.persist` is called inside `Loop.propose()` before `review()`. For fal.ai (returns URL, downloading is expensive) or any provider that emits a deferred reference, this forces eager fetch. The user might REJECT and never need the bytes.

**Why it matters:** Cost. fal.ai image generation can be GB-sized; downloading every reject is wasteful. Real adopters will work around it; the workaround should be in the protocol.

**Recommended fix:** Already covered in F3. Same shape: `stage` / `commit` / `discard`, or optional `materialize`.

---

### F10 — `extra_params` has no schema discipline

**Severity:** moderate
**Area:** loop-state-machine / log-schema

**Issue:** `extra_params: dict[str, Any]` is the right shape for "anything the generator wants to remember," but there's no way for an adapter or generator to declare what it expects. Replay tooling must trust that the dict on a 6-month-old log entry still makes sense to today's adapter.

**Why it matters:** Determinism enforcement (S7) wants to validate "if I replay this entry, will the same parameters even apply to today's adapter?" Without a schema, validation is a vibes check.

**Recommended fix:** Optional `Adapter.params_schema -> dict | None` returning a JSON Schema. If declared, replay validates the entry's `extra_params` against it; if not, replay proceeds and warns. Not v0.1.0 work — flag as v0.6.0 (where it pairs naturally with determinism enforcement).

---

### F11 — Single-int seed assumption

**Severity:** moderate
**Area:** extraction-readiness

**Issue:** `LogEntry.seed: int | None` assumes one integer. fal.ai's flux-pro uses a single int seed (fine). Stable Diffusion (sampler+model+CLIP seeds — three ints) doesn't fit. Multi-step pipelines (concept → 3D → texture, three different seeds) don't fit.

**Why it matters:** First non-text adapter that hits this lies (encodes one of the seeds, drops the others).

**Recommended fix:** Cheapest path: document that "seed" means "the principal seed; nested seeds go in `extra_params`." Best path: `seed: int | str | dict | None`. Either is acceptable; pick before v0.2.0 ships an Anthropic generator (which itself doesn't honor seeds and will reveal whether the field even applies).

---

### F12 — No session lineage for revised artifacts

**Severity:** moderate
**Area:** log-schema

**Issue:** `Log.is_resolved` ends a session on first ACCEPT. A revised artifact (Nassau v1 ships, six months later we revisit with new vegetation rules) is necessarily a *new session*, but the new log has no link back to its predecessor. The authorship trail breaks at session boundaries.

**Why it matters:** "The artifact is reproducible from the log alone" is true for one session, but in practice artifacts evolve across sessions. Without `parent_session_id`, the v2 log has no way to say "I started from v1's accepted output."

**Recommended fix:** Add `Log.parent_session_id: str | None = None` now. Cost is negligible (one optional field, one YAML key, one `from_dict` line). Retrofitting it post-v1.0 would either break replay tools or require a lossy migration.

### F14 — `supports_seed = True` on EchoGenerator is misleading

**Severity:** minor
**Area:** API-surface / determinism-contract

**Issue:** EchoGenerator's "supports seed" means "the seed value appears in the deterministic output," not "the seed is used as RNG." A reviewer of an Echo log entry sees `seed=42, supports_seed=True` and infers "this output is reproducible against a seed-honoring model" — true here, but the protocol's intent is unclear.

**Recommended fix:** Either rename `supports_seed → honors_seed` (clearer), or document explicitly that "supports" means "incorporates into output," not "RNG-seeds." Trivial work; do it before external consumers code against the name.

## Strengths worth keeping

- **Adapter/Generator separation is the right cleavage.** "Generator owns the provider, Adapter owns the artifact type" is exactly correct. Many naïve designs collapse these into one. Lock this in.
- **Loop's pending-state machine.** Forcing review before next propose at the type level is the methodology encoded as code. Don't relax this — F4's "batch" suggestion preserves the discipline; relaxing it further would not.
- **Append-only Log via `iteration` + `parent_id` invariants.** The chain check on `append()` (`expected_iteration`, `expected_parent_id`) is robust and catches the common bug ("I edited an old entry"). Keep.
- **Frozen dataclasses for `LogEntry` + `ModelIdentity`.** Immutability of logged entries is non-negotiable for an authorship trail. Keep.
- **`@runtime_checkable` Protocols (not ABCs).** Lets adapters duck-type without forced inheritance — the right call for an extraction-friendly package. Keep.
- **`SEMANTIC_B` as the default tier.** Honest about model deprecation reality. Keep.
- **`StrEnum` for Verdict and DeterminismTier.** Round-trips cleanly through YAML without custom serializers. Keep.
- **Clear ABANDON verdict.** Most prior-art critique frameworks force a binary accept/reject and lose "the whole approach is wrong." ABANDON is right.
- **Test coverage at the protocol layer (`test_protocols.py`).** Testing that EchoGenerator/TextFileAdapter actually conform to their Protocols is exactly the discipline that v0.5.0's vegetation adapter will need. Generalize this in F7.
- **Honest pre-alpha framing.** README is upfront: "expect breaking changes until v1.0." This permits the F1-F5 changes proposed above without violating any public commitment.

## Roadmap recommendations

1. **Pull F5 (determinism negotiation) into S3 / v0.2.0.** S3-3 is "Determinism Tier A/B/C declared per adapter" — that's the wrong shape per F5. Reframe to "Determinism Tier negotiated by Loop from Adapter + Generator + user request." Cost: one extra ticket, paid before any external generator ships.

2. **Add a v0.1.5 "Protocol hardening" sprint between S2 and S3.** Pulls F1 (GenerationRequest/Result), F2 (SemVer schema), F3/F9 (stage/commit/discard), F12 (parent_session_id). This is the *cheap* moment to do this work — there is exactly one consumer (text adapter) and zero external pressure. Once S3's Anthropic generator ships, every change to these shapes is a breaking change. The 9-sprint roadmap can absorb a 0.5 sprint here without disturbing v1.0 timing.

3. **Promote the protocol-conformance harness (F7) into S3.** v0.2.0 is going to write the first non-trivial generator (Anthropic). It should ship with the harness it'll use to validate itself. Backports cleanly to TextFileAdapter/EchoGenerator.

4. **Don't move S6 (vegetation dogfood) earlier.** It is correctly placed at v0.5.0 — needs S3-S5 to mature first. But pull S6's *signature exploration* forward as a thought experiment in S3: write the `VegetationDensityAdapter` skeleton against current protocols and see what breaks. That's the cheapest possible dogfood signal.

5. **S8's "API freeze candidate" sprint should bake-in F2 and F12.** Once SemVer-versioned schema and `parent_session_id` are in, the freeze is meaningful. Without them, the freeze is on a schema that is going to break.

## Risks to surface

- **Provider reality vs. determinism contract.** Today's frontier LLMs (Claude, GPT-4o) do not honor seeds in practice, even when they accept the parameter. STRICT_A as a *defaultable* tier risks making CDD's main promise feel unfulfillable. Consider whether STRICT_A should require an explicit opt-in *with* a generator capability check, rather than being a tier the user can request unconditionally. F5 above covers the mechanism; the framing matters too.

- **The v0.5.0 dogfood gate is the real validation.** Until a 4096×4096-array vegetation adapter exists and runs through the Loop, the protocols are theoretical. The roadmap correctly schedules this; the architectural review can only catch obvious holes. Real holes will surface in S6 — budget time in S7/S8 to fold the resulting changes.

- **CLI surface area (S4) hasn't been designed yet and will exert heavy pressure on the protocols.** `cdd review` is interactive — that's a different shape than the synchronous propose/review pair the Loop assumes. F4's batch question and F1's streaming question both lurk here. Spend an hour sketching the CLI flow before S3 starts; it will surface protocol pressure earlier.

- **YAML editability is a feature *and* a hazard.** "Logs are human-editable" is part of the pitch; it's also a way to silently break the chain (mismatched parent_ids, wrong iteration numbers). Once a log is edited by hand, all replay guarantees are voided. Document this explicitly: edit a log = give up replay. Or add a `Log.verify_chain()` integrity check that downstream tools call before replay.

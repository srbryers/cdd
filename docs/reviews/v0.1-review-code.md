# CDD v0.1.0 — Round 1 Code Review

## Verdict: APPROVE-WITH-CHANGES

The v0.1.0 abstractions are well-shaped, well-documented, and well-tested. The protocol/adapter/generator/loop separation is clean and the YAML round-trip is correct in the happy path. The findings below are concrete bugs and edge-case gaps that I'd want to see folded in before tagging this as the "protocol surface that downstream consumers will build against," but none of them block merging the PR — they're better as a follow-up commit on the same branch or a tight v0.1.1.

The two findings I would NOT ship without are F1 (`read_log` on empty/null YAML files) and F4 (CLI stub contradicts its own docstring). The first is a one-liner that prevents a confusing AttributeError; the second is a documentation lie that gets baked into a public release.

## Detailed findings

### F1 — `read_log` of empty/null YAML crashes with the wrong exception

**Severity:** major
**File:** src/cdd/log.py:200-203 (`read_log` → `from_dict`)
**Issue:** `yaml.safe_load("")` returns `None`. `from_dict(None)` then calls `data.get("cdd_log_version")` and raises `AttributeError: 'NoneType' object has no attribute 'get'`. The same happens for a file containing only `null` or whitespace, or a YAML document that's a top-level scalar (e.g., the user accidentally wrote a string).
**Why it matters:** `read_log` advertises `LogSchemaError` and `FileNotFoundError` as its failure modes. Surfacing `AttributeError` instead gives downstream callers nothing useful to catch and breaks the contract advertised in the docstring. This is exactly the kind of silent footgun new users hit when they hand-create their first log file.
**Recommended fix:**
```python
def from_dict(data: dict[str, Any]) -> Log:
    if not isinstance(data, dict):
        raise LogSchemaError(
            f"expected a YAML mapping at top level, got {type(data).__name__}"
        )
    version = data.get("cdd_log_version")
    if version != LOG_SCHEMA_VERSION:
        raise LogSchemaError(...)
    ...
```
And add a test:
```python
def test_read_log_empty_file_raises_schema_error(tmp_path: Path) -> None:
    p = tmp_path / "empty.yaml"
    p.write_text("", encoding="utf-8")
    with pytest.raises(LogSchemaError):
        read_log(p)
```

### F2 — `from_dict` does not replay chain validation

**Severity:** major
**File:** src/cdd/log.py:152-173
**Issue:** The Log class's invariants (strictly-incrementing iteration, parent_id chains, no append-after-resolved) are enforced by `Log.append`. But `from_dict` constructs the log by passing a pre-built `entries` list directly to the dataclass constructor, bypassing `append`. A hand-edited or corrupted YAML can therefore produce a `Log` instance with a broken chain that `Loop` then uses as if it were valid.

Concrete example: a YAML with iterations `[1, 2, 4]` (skipping 3) loads cleanly. `Loop(log=...)` then calls `propose()`, `review()`, and the new entry's iteration is `len(entries)+1 = 4` — but the existing log already has iteration 4. `Log.append` will raise (because expected = 4, got = 4) — wait, no, it expects `len(entries)+1 = 4`, the new entry is iteration 4 — match. So actually the corruption cascades silently because the new entry inherits the broken numbering scheme.

Worse: if `parent_id` is corrupted mid-chain, `read_log` accepts it; the next `append` only validates against the LAST entry, not the whole chain.
**Why it matters:** This is the protocol surface CDD claims for authorship reproducibility. If a corrupted log can be loaded, modified, and re-written, the authorship trail's integrity guarantee is conditional on no one ever hand-editing a file. Given YAML is human-editable on purpose, that's a contradiction.
**Recommended fix:** Reconstruct the log via `append`:
```python
def from_dict(data: dict[str, Any]) -> Log:
    ...
    log = Log(
        cdd_log_version=version,
        session_id=data["session_id"],
        artifact_type=data["artifact_type"],
        determinism_tier=DeterminismTier(data["determinism_tier"]),
        created_at=datetime.fromisoformat(data["created_at"]),
    )
    for entry_data in data["entries"]:
        log.append(_entry_from_dict(entry_data))
    return log
```
This makes `LogConsistencyError` part of `read_log`'s failure surface — document it.

### F3 — Path traversal in TextFileAdapter

**Severity:** major
**File:** src/cdd/adapters/text.py:58-77 (persist), 79-81 (load)
**Issue:** `persist` interpolates `namespace` directly into a path: `f"{namespace}/{digest}.txt"`. A namespace of `"../foo"` resolves to a sibling of `base_dir`. Likewise `load(output_ref)` accepts arbitrary `output_ref` strings — a malicious or hand-edited log with `output_ref: "../../etc/passwd"` causes `read_text` to read outside `base_dir`.

The session_id default is UUID4 (safe), but a custom `session_id` (which becomes the namespace) can come from anywhere.
**Why it matters:** Two real concerns. (a) Operational: a developer experimenting with custom session IDs gets a confusing "where did my files go" bug. (b) When CDD logs are shared between projects (the README pitches "the artifact is reproducible from the log alone"), `load(output_ref)` on a foreign log becomes a trust-boundary read.
**Recommended fix:**
```python
import os

def _resolve_under(self, rel: str) -> Path:
    full = (self._base / rel).resolve()
    base = self._base.resolve()
    if not full.is_relative_to(base):  # 3.9+
        raise ValueError(f"path {rel!r} escapes base_dir {base}")
    return full

def persist(self, raw_output, *, namespace):
    if "/" in namespace or namespace in ("", ".", "..") or namespace.startswith("."):
        raise ValueError(f"invalid namespace: {namespace!r}")
    ...
    full_path = self._resolve_under(rel_path)
    ...
```
And add tests: `test_persist_rejects_namespace_with_separator`, `test_load_rejects_traversal_ref`.

### F6 — Persist failure after a successful generate silently discards the output

**Severity:** moderate
**File:** src/cdd/loop.py:180-198
**Issue:**
```python
raw_output = self._generator.generate(prompt, ...)   # may have cost real money
output_ref = self._adapter.persist(raw_output, ...)   # may raise (disk full, perm denied, etc.)
self._pending = _Pending(...)
```
If `persist` raises, the loop state is clean (no pending) — but the generator already produced output. For paid LLM/image API generators, this is a real loss. There's no recovery path; the user has to call `propose` again, which calls `generate` again.
**Why it matters:** Once CDD has real Anthropic/fal.ai generators (v0.2.0), this becomes a wallet-level concern. Worth fixing the foundation now.
**Recommended fix:** Either:
1. Catch persist failures and surface them with the raw output attached for caller-side recovery: `raise LoopPersistError("...", raw_output=raw_output)`.
2. Allow proposing without persisting (split the operation into `generate` + `persist`/`stage`).

For v0.1.0, option 1 is sufficient and cheap. Document the contract: "if `persist` fails, the raw output is on the exception; the loop remains in a clean state ready for retry."

### F7 — Schema-mismatch error is confusing when the version key is missing entirely

**Severity:** moderate
**File:** src/cdd/log.py:159-164
**Issue:** A YAML file that's well-formed but has no `cdd_log_version` key (e.g., a half-finished hand-edited log, or a YAML from some other tool) produces `LogSchemaError: unsupported cdd_log_version: None`. "None" is technically correct but the user-readable hint should be "this doesn't look like a CDD log."
**Why it matters:** Error messages are the documentation read most often by people in trouble. A misleading one wastes ten minutes.
**Recommended fix:**
```python
version = data.get("cdd_log_version")
if version is None:
    raise LogSchemaError(
        "missing cdd_log_version — this doesn't appear to be a CDD log file"
    )
if version != LOG_SCHEMA_VERSION:
    raise LogSchemaError(
        f"unsupported cdd_log_version: {version!r} "
        f"(this CDD speaks version {LOG_SCHEMA_VERSION})"
    )
```

### F9 — Conflicting `session_id` and `log` arguments are silently reconciled

**Severity:** moderate
**File:** src/cdd/loop.py:71-117
**Issue:** `Loop(adapter=..., generator=..., session_id="my-session", log=existing_log_with_different_id)` — the `session_id` argument is ignored without warning. The docstring says "Ignored when log is provided," which is fine, but the silent override is the kind of thing that wastes an afternoon.
**Why it matters:** Resume flows are exactly where developers paste boilerplate from a non-resume flow; the failure is "everything works but the session ID isn't what I asked for."
**Recommended fix:**
```python
if log is not None and session_id is not None and session_id != log.session_id:
    raise LoopStateError(
        f"session_id {session_id!r} conflicts with log.session_id {log.session_id!r}; "
        "drop one"
    )
```

### F10 — `Log.append` doesn't enforce id uniqueness or self-reference

**Severity:** minor
**File:** src/cdd/log.py:105-131
**Issue:** `append` validates iteration number and parent_id against the previous entry only. It doesn't check:
- `entry.id != entry.parent_id` (self-reference)
- `entry.id` is not already in `self.entries` (duplicate)
**Why it matters:** Hand-built `LogEntry` instances or YAML round-trips of malformed data could pass the existing checks. Cheap to add; closes the integrity loop.
**Recommended fix:** Two extra checks at the top of `append`. The set lookup is O(N) per append, fine at scale.

### F11 — `to_dict` and `from_dict` are not in the package public API

**Severity:** minor
**File:** src/cdd/__init__.py
**Issue:** `LOG_SCHEMA_VERSION` is exported from `cdd.__init__`. `to_dict`/`from_dict` are not — they're reachable via `from cdd.log import ...`. Anyone consuming the schema version constant is plausibly also serializing schema, so this is asymmetric.
**Why it matters:** v0.1.0 is the version downstream consumers pin against. Setting the surface now is cheaper than expanding it later.
**Recommended fix:**
```python
from cdd.log import (
    LOG_SCHEMA_VERSION,
    Log,
    LogConsistencyError,
    LogSchemaError,
    from_dict,
    read_log,
    to_dict,
    write_log,
)
```
And add to `__all__`.

### F12 — Hash collision claim is correct but understated

**Severity:** minor
**File:** src/cdd/adapters/text.py:39-42
**Issue:** Comment says "16 hex chars is enough to make collisions extraordinarily unlikely within a single session." The actual property: collision is bounded by namespace, sessions don't share namespaces by default, so the claim is correct. The comment just doesn't say *why* it's true.
**Recommended fix:** Tighten the docstring:
```
"""How many hex chars of SHA-256 to use in the filename. 16 hex chars (64 bits)
gives birthday-bound collision resistance over ~4B entries within one namespace.
The Loop uses session_id as namespace, so cross-session collisions are not
possible by default."""
```

### F13 — One test uses manual try/except instead of `pytest.raises`

**Severity:** minor
**File:** tests/test_protocols.py:67-73
**Issue:**
```python
def test_adapter_rejects_wrong_type() -> None:
    a = _MinimalTextAdapter()
    try:
        a.persist(b"bytes-not-str", namespace="loop-42")
    except TypeError:
        return
    raise AssertionError("expected TypeError on wrong raw_output type")
```
This works, but it (a) returns from the test on success rather than asserting, (b) doesn't pin the exception message, (c) is inconsistent with the rest of the suite (which uses `pytest.raises`).
**Recommended fix:**
```python
def test_adapter_rejects_wrong_type() -> None:
    a = _MinimalTextAdapter()
    with pytest.raises(TypeError, match="str"):
        a.persist(b"bytes-not-str", namespace="loop-42")
```

### F14 — No test for empty-string artifact

**Severity:** minor
**File:** tests/test_text_adapter.py
**Issue:** `TextFileAdapter.persist("", namespace="x")` works (writes 0-byte file at SHA-256 of empty string prefix). No test covers it. Small but it's exactly the kind of edge case that breaks under future refactors (someone adds `assert raw_output, "expected non-empty"` to "be safe").
**Recommended fix:**
```python
def test_persist_empty_string_round_trips(tmp_path: Path) -> None:
    a = TextFileAdapter(tmp_path)
    ref = a.persist("", namespace="ns")
    assert a.load(ref) == ""
```

### F15 — Dataclass + append-discipline is a soft contract

**Severity:** minor
**File:** src/cdd/log.py
**Issue:** `Log` is a regular (non-frozen) dataclass with a `field(default_factory=list)` for `entries`. Anyone can `log.entries.append(...)` directly, bypassing `Log.append` validation. The module docstring says "read code should never mutate `entries`" but that's a comment, not enforcement.
**Why it matters:** Tied to F2 — if `from_dict` doesn't replay through `append`, and direct list-mutation also bypasses `append`, then `Log.append` is the only path that enforces invariants and three other paths exist.
**Recommended fix:** Either (a) make `entries` a tuple internally and rebuild on append (slight overhead), (b) add a `_validate_chain()` method called from constructors / round-trip, or (c) explicitly accept this as "Log is a dataclass and append is the discipline" — which is fine for v0.1, just be deliberate.

### F16 — Defensive double-copy of `extra_params`

**Severity:** minor
**File:** src/cdd/loop.py:196 + 226
**Issue:** `propose()` copies `extra_params` into `_Pending` (line 196). `review()` copies it again into the `LogEntry` (line 226). Since `_Pending` is internal and never exposed, the second copy is unnecessary.
**Recommended fix:** Drop one copy. Or add a comment explaining the redundancy is intentional (e.g., "frozen LogEntry must have a private dict; defensive copy here matches that frozen contract"). Pick one and document.

## Strengths worth keeping

- **Clean two-stage Loop lifecycle (`propose` then `review`)** — makes the discipline ("nothing is logged that wasn't reviewed") enforceable in code, not in convention. The `_Pending` internal state is exactly the right shape.
- **Runtime-checkable Protocols + duck-typed reference impls** — `TextFileAdapter` and `EchoGenerator` don't inherit from `Adapter`/`Generator`, and the tests assert `isinstance(a, Adapter)` works anyway. That's the right idiom for v0.1 protocol design.
- **`LOG_SCHEMA_VERSION` constant + `LogSchemaError`** — the schema gate is in place from day one, even if its policy needs a small fix (F7). Bumping the version later is straightforward.
- **Resume-from-log flow with mismatch checks** — the artifact_type / determinism_tier checks at `Loop.__init__` (lines 100-109) are exactly the consistency checks needed; both have tests.
- **Frozen `LogEntry`** — immutability for log entries is non-negotiable for an authorship trail; this nails it. The honest call-out about `extra_params` dict mutability is the right level of disclosure.
- **Content-addressed `output_ref`** — the dedup property for SEMANTIC_B re-rolls is genuinely useful and the docstring explains it.
- **Tests cover the chain invariants thoroughly.** `Log.append` has explicit tests for parent_id mismatch, iteration mismatch, post-resolved append, both ACCEPT and ABANDON. Solid.
- **End-to-end integration test exercises the real adapter and generator on the real filesystem** — not just stubs. That's what catches integration drift.
- **YAML output is verified to be human-readable** by `test_yaml_output_is_human_readable` — asserting on substrings of the rendered file. Good "the docs claim X; the test enforces X" pairing.

## Notable design choices flagged for architect review (not code review)

- **CLI version target.** F4 surfaces a documentation contradiction. The deeper question is *which* version should ship the CLI. v0.1.0 is the "abstractions" release; CLI in v0.1 would be premature. v0.3.0 is plausible. But "init / review / replay / diff" are core to the methodology pitch in the README — without them, CDD is a Python library, not a methodology framework. This is an architect call.

- **Determinism tier negotiation between Adapter and Generator.** The Adapter declares its tier; the Generator declares `supports_seed`. The Loop wires them but doesn't reconcile: a STRICT_A adapter paired with a `supports_seed=False` generator is a configuration error that's not caught. The docstring on `Generator.supports_seed` says "the Loop should warn or refuse strict mode" but the Loop currently does neither. v0.1.0 ships SEMANTIC_B by default so this isn't biting; v0.2.0 with real generators will hit it.

- **`Log` mutability vs `LogEntry` immutability.** `LogEntry` is frozen, `Log` is not. Pragmatic for the append-during-session use case. But it makes the round-trip story (read_log → load → modify → write_log) ambiguous about what's "allowed." Document it explicitly; F15 is the same observation from a code-quality angle.

- **`output_ref` is opaque to CDD.** The protocol says "the adapter interprets this." That's correct for flexibility but loses the ability for CDD itself to validate refs (F3 path traversal sneaks in because of this). A future option: CDD-defined ref schemes (`file://`, `sha256:`, `s3://`) with an opt-out for adapters that need full opacity.

- **Single-pending-proposal discipline.** The Loop holds at most one pending proposal. This is the right v0.1 default — it forces the human to review every output. But it forecloses a "generate 5, pick 1" UX that some real critique flows actually want. Worth naming as a deliberate choice; v0.2+ might want a `LoopBatch` companion type that holds N pending proposals and reviews them in any order.

- **Generator failure modes (F6).** Real generators (Anthropic, fal.ai) will raise occasionally. The Loop currently lets exceptions surface unchanged from `propose` — clean state, no pending, no log entry. That's correct for transient failures (retry the call). It's wrong for the persist-after-generate failure mode where the cost has been incurred. v0.2.0 generators need a clearer contract here.
